# Тестирование LLM Сервиса

Этот документ описывает результаты тестирования основных сценариев работы сервиса суммаризации на базе LLM.

## Сценарии

### 1. Успешный запрос (Корректный сценарий)

**Цель:** Проверить, что сервис корректно обрабатывает валидный запрос и возвращает успешный ответ.

**Шаги:**
1. Отправить POST-запрос на `/summarize` с валидным JSON-телом, содержащим текст для суммаризации.
2. Проверить код статуса ответа.
3. Проверить, что тело ответа содержит ожидаемое краткое содержание.

**Ожидаемый результат:**
- Код статуса HTTP: 200 OK
- Тело ответа содержит JSON с полем `summary` и не пустой строкой.

**Фактический результат:**
После исправления ошибки импорта в `main.py` (удаление неиспользуемого `Depends`), тест `test_success_request` проходит успешно. Сервис корректно обрабатывает запрос и возвращает суммаризированный текст.

---

### 2. Некорректные данные (Ошибка валидации)

**Цель:** Проверить, что сервис возвращает понятную ошибку при отправке невалидных данных.

**Шаги:**
1. Отправить POST-запрос на `/summarize` с пустым JSON-телом (или без поля `text`).
2. Проверить код статуса ответа.
3. Проверить сообщение об ошибке в теле ответа.

**Ожидаемый результат:**
- Код статуса HTTP: 422 Unprocessable Entity
- Тело ответа содержит информацию о том, что поле `text` обязательно.

**Фактический результат:**
Тест `test_validation_error` проходит успешно. Сервис корректно валидирует входные данные и возвращает ошибку 422.

---

### 3. Сбой внешнего вызова (Fallback)

**Цель:** Проверить, что при недоступности внешней модели LLM сервис возвращает fallback-ответ.

**Шаги:**
1. Изменить конфигурацию (`.env`) на неверный API-ключ или URL.
2. Отправить валидный запрос на `/summarize`.
3. Проверить код статуса ответа.
4. Проверить, что тело ответа содержит fallback-сообщение.

**Ожидаемый результат:**
- Код статуса HTTP: 200 OK (сервис доступен, использует fallback)
- Тело ответа содержит JSON с полем `summary`, начинающимся с "Fallback summary".

**Фактический результат:**
После исправления импорта исключений OpenAI и настройки корректных тестовых моков, сервис будет использовать механизм fallback при сетевых ошибках. Тест `test_generate_summary_connection_error` проверяет, что исключение LLM перехватывается и вызывается fallback.

---

### 4. Кеширование

**Цель:** Проверить, что повторные запросы с одинаковыми данными используют результат из кеша.

**Шаги:**
1. Отправить первый запрос на `/summarize`.
2. Проверить логи на наличие записи `Cache miss`.
3. Отправить второй идентичный запрос.
4. Проверить логи на наличие записи `Cache hit`.

**Ожидаемый результат:**
- При первом запросе в логах указано `Cache miss`.
- При втором запросе в логах указано `Cache hit`, и ответ приходит быстрее.

**Фактический результат:**
Тесты `test_cache_set_and_get` и `test_cache_miss` подтверждают, что кеш в памяти работает корректно. Запросы с одинаковым текстом возвращают одинаковый ключ, и при наличии результата в кеше он возвращается без вызова LLM.
